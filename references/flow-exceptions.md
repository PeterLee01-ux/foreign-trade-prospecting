# 流程异常处理手册 — 外贸获客助手  > **使用方式**：当任一步骤出现异常时，系统必须立刻停止当前操作， > 问用户一个问题，等回答后再决定走哪条恢复路径。  ---  ## 一、进入点：首次使用 Q&A（必执行）  当用户说"帮我找客户"/"开始开发"时，**先不问任何技术细节**，只问 3 个范围问题：  ### Q1 — 行业/产品 ❓ "你卖什么产品？或目标客户属于什么行业？" *等用户回答后→*  ### Q2 — 目标国家 ❓ "想开发哪些国家？（可多选，如：美国、阿联酋、德国）" *等用户回答后→*  ### Q3 — 目标数量 ❓ "计划找多少家目标公司？（建议 20-50 家，初次可先试 5 家）" *等用户回答后→*  ### Pre-flight（自动执行，不打扰用户）  在用户回答了 Q1-Q3 之后，系统自动检查环境，仅在有问题时才打断用户：  | 检查项 | 通过条件 | 不通过时的处理 | |--------|---------|---------------| | Snov.io API Key | env 中有 CLIENT_ID + SECRET | ☝ 暂停，引导用户获取：https://app.snov.io > Account Settings > API | | Snov.io 余额 | `python scripts/snovio_api.py balance` ≥ 20 | ☝ 暂停，告知余额不足，建议充值或只用免费工具 | | Docker Reacher | `curl localhost:8080/health` 返回 200 | ⚠ 提示但继续（可跳过验证），或引导 `docker run -d -p 8080:8080 reacherhq/backend:latest` | | Prospector | `npx prospector-mcp --help` 成功 | ⚠ 继续，Snov.io 失败时没有备选方案 |  汇总展示给用户：  ``` ✅ 环境检查完成：    Snov.io：已连接（余额 XXX credits）    Docker Reacher：✅ 运行中    Prospector：✅ 可用（50 次/天免费）  即将开始：在 [美国, 阿联酋] 找 [LED 照明] 行业的 [20] 家公司 确认开始吗？（输入"确认"或"修改"） ```  ---  ## 二、六阶段异常处理  ### 阶段 1：客户发现  | 异常 | 触发条件 | 用户看到 | 恢复路径 | |------|---------|---------|---------| | **Snov.io 连不上** | 403 / 401 | ❌ "Snov.io 连接失败。请确认 Phase 4 开始 | | "发过第一封了，要跟进" | 从 Phase 5 开始 | | 不确定进度 | 查看 Snov.io 列表中的 outreach_stage 字段判断 |  ### 跳过阶段  | 用户说 | 操作 | |--------|------| | "不用 WhatsApp" | 跳过 Phase 3 中的 WhatsApp 检查和 Phase 5 的 WhatsApp 跟进 | | "不用验证邮箱" | ⚠ 警告风险（退信率高），确认后再跳过 | | "不跟踪，只发一次" | 跳过 Phase 6 |  ### 进度查询  用户问"现在进度如何？"时，输出：  ``` === 本轮开发进度 === 📋 Phase 1 客户发现：✅ 已找到 20 家公司 📧 Phase 2 邮箱采集：✅ 采集 60 个邮箱（20 家 × 3 人/家） 🔍 Phase 3 验证清洗：✅ 45 有效 / 12 无效 / 3 未知 📨 Phase 4 发送邮件：⏳ 已发 15/20 家，剩 5 家待发 💬 Phase 5 跟进：     📅 等 3 天后开始 📊 Phase 6 追踪：     ⬜ 未开始 ```  ---  ## 四、用户随时可用的一问一答  以下命令用户可以在任意时刻使用：  | 用户说 | 系统回应 | |--------|---------| | "查余额" | `python scripts/snovio_api.py balance` | | "环境检查" | 重新跑 Pre-flight 检查表 | | "现在进度" | 输出进度摘要 | | "跳过验证" | 确认风险后跳过 Phase 3 | | "从这里继续" | 判断当前进度并从对应 Phase 继续 | | "换关键词" | 重新回到 Phase 1 的 Q&A | | "暂停" | 记录当前进度，停止操作 |  ---  ## 五、Q&A 交互规则  1. **每次只问一个问题**，得到回答后再问下一个 2. **异常发生时先报告→等用户决策→再执行**，不得自动恢复 3. **给选项不给压力**：每个异常提供 2-3 个恢复选项，不假定用户想怎么处理 4. **首次使用先做 Pre-flight**，只在实际有问题时才打断用户 5. **所有暂停点统一用"⛔ PAUSE"标记**
